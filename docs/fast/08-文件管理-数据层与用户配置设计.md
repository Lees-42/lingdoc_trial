## 1. 数据层概述

### 1.1 核心数据表

文件管理模块的数据层由 ** Vault 文件索引表** 和 **AI 元数据表** 两张主表构成，通过 `file_id` 1:1 关联。

#### 1.1.1 Vault 文件索引表 `lingdoc_file_index`

```sql
-- Vault 文件索引表
DROP TABLE IF EXISTS `lingdoc_file_index`;
CREATE TABLE `lingdoc_file_index` (
  `file_id`         varchar(64)   NOT NULL COMMENT '文件唯一ID（UUID）',
  `user_id`         bigint(20)    NOT NULL COMMENT '所属用户ID',
  `file_name`       varchar(256)  NOT NULL COMMENT '文件名（含扩展名）',
  `vault_path`      varchar(512)  NOT NULL COMMENT 'Vault内相对路径',
  `abs_path`        varchar(512)  NOT NULL COMMENT '绝对路径',
  `file_type`       varchar(32)   NOT NULL COMMENT '文件类型',
  `file_size`       bigint(20)    NOT NULL COMMENT '文件大小（字节）',
  `checksum`        varchar(64)   NOT NULL COMMENT 'SHA256',
  `sub_path`        varchar(512)  DEFAULT NULL COMMENT '子分类路径（相对于vault_root/documents/）',
  `source_type`     char(1)       DEFAULT '0' COMMENT '来源：0手动上传 1自动规整 2表格助手生成',
  `file_content`    longtext      DEFAULT NULL COMMENT '文件文本内容',
  `content_path`    varchar(512)  DEFAULT NULL COMMENT '大文件文本内容的外部存储路径（分层存储预留）',
  `content_size`    bigint        DEFAULT 0 COMMENT '文本内容字节数',
  `is_desensitized` char(1)       DEFAULT '0' COMMENT '是否有脱敏副本：0否 1是',
  `create_time`     datetime      DEFAULT CURRENT_TIMESTAMP,
  `update_time`     datetime      DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`file_id`),
  KEY `idx_user_id` (`user_id`),
  KEY `idx_file_type` (`file_type`),
  KEY `idx_source_type` (`source_type`),
  KEY `idx_checksum` (`checksum`),
  UNIQUE KEY `uk_user_checksum` (`user_id`, `checksum`),
  KEY `idx_user_type_time` (`user_id`, `file_type`, `create_time`),
  FULLTEXT KEY `ft_name_content` (`file_name`, `file_content`) WITH PARSER ngram
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='Vault文件索引表';
```

> **说明**：`tag` 字段已从本表删除。标签不再与目录层级绑定，改为独立的标签体系（见第2节）。`file_content` 保留用于 MVP 阶段统一存储，未来可启用分层存储（见第4节）。

#### 1.1.2 文件 AI 元数据表 `lingdoc_file_ai_meta`（脚手架）

AI 相关字段**不混入** `lingdoc_file_index`，单独建表，由 AI 开发同学后续完善设计。

```sql
-- 文件 AI 元数据表（脚手架，供AI开发同学完善）
DROP TABLE IF EXISTS `lingdoc_file_ai_meta`;
CREATE TABLE `lingdoc_file_ai_meta` (
  `file_id`          varchar(64)   NOT NULL COMMENT '文件ID（= lingdoc_file_index.file_id），1:1关联',
  `kb_id`            varchar(64)   DEFAULT NULL COMMENT '所属知识库ID（NULL表示未入知识库）',
  `parse_status`     char(1)       DEFAULT '0' COMMENT '文本解析状态：0待处理 1处理中 2已完成 3失败',
  `parse_error`      varchar(512)  DEFAULT NULL COMMENT '解析错误信息',
  `chunk_count`      int           DEFAULT 0 COMMENT '文本分块数量',
  `embedding_status` char(1)       DEFAULT '0' COMMENT '向量化状态：0未处理 1处理中 2已完成 3失败',
  `summary`          varchar(512)  DEFAULT NULL COMMENT 'AI一句话概括',
  `keywords`         varchar(1024) DEFAULT NULL COMMENT 'AI提取关键词（JSON数组）',
  `create_time`      datetime      DEFAULT CURRENT_TIMESTAMP,
  `update_time`      datetime      DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`file_id`),
  KEY `idx_kb_id` (`kb_id`),
  KEY `idx_parse_status` (`parse_status`),
  KEY `idx_embedding_status` (`embedding_status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='文件AI元数据表（脚手架）';
```

**用途**：
- `summary`：用户检索时联表查询，帮助快速识别文件内容
- `keywords`：AI RAG 检索时的辅助检索维度
- `kb_id IS NOT NULL`：表示该文件已加入知识库
- `parse_status = '2'`：文本提取已完成
- `embedding_status = '2'`：向量化已完成

#### 1.1.3 关联表

| 表名 | 关系 | 说明 |
|------|------|------|
| `lingdoc_file_version` | 1:N ← `file_id` | 版本溯源（FAST-007 已定义） |
| `lingdoc_file_ai_meta` | 1:1 = `file_id` | AI 处理元数据（脚手架） |
| `lingdoc_form_reference` | N:1 → `doc_id`(`file_id`) | 表格助手参考文档关联 |
| `kb_chunk` | N:1 → `file_id` | 文本分块（外键改为 `file_id`） |
| `kb_knowledge_base` | 1:N → `kb_id` | 知识库配置表（保留） |

## 2. 标签体系

### 2.1 设计原则

- **标签与目录解耦**：`sub_path` 仅表示物理/逻辑位置，标签是独立的分类维度。
- **目录可打标签**：目录（文件夹路径）可以绑定标签，其子文件和子目录自动继承。
- **文件可独立打标签**：文件除继承父目录标签外，还可有自己的独立标签。
- **继承可视化**：继承标签在 UI 中以虚线/灰色样式展示，直接绑定标签以实线/彩色展示。

### 2.2 标签定义表 `lingdoc_tag`

```sql
DROP TABLE IF EXISTS `lingdoc_tag`;
CREATE TABLE `lingdoc_tag` (
  `tag_id`      varchar(64)   NOT NULL COMMENT '标签唯一ID（UUID）',
  `tag_name`    varchar(128)  NOT NULL COMMENT '标签名（如"奖学金"、"操作系统"）',
  `tag_color`   varchar(32)   DEFAULT '#409EFF' COMMENT '标签颜色（前端展示用）',
  `tag_scope`   char(1)       DEFAULT 'A' COMMENT '适用范围：A所有 F仅文件 D仅目录',
  `sort_order`  int           DEFAULT 0 COMMENT '排序号',
  `create_time` datetime      DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`tag_id`),
  UNIQUE KEY `uk_tag_name` (`tag_name`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='标签定义表';
```

### 2.3 标签绑定表 `lingdoc_tag_binding`

```sql
DROP TABLE IF EXISTS `lingdoc_tag_binding`;
CREATE TABLE `lingdoc_tag_binding` (
  `binding_id`  varchar(64)   NOT NULL COMMENT '绑定记录ID',
  `target_type` char(1)       NOT NULL COMMENT '目标类型：F文件 D目录',
  `target_id`   varchar(512)  NOT NULL COMMENT '目标标识：文件时为file_id，目录时为目录路径',
  `tag_id`      varchar(64)   NOT NULL COMMENT '标签ID',
  `bind_type`   char(1)       DEFAULT '0' COMMENT '绑定类型：0直接绑定 1继承自父目录',
  `create_time` datetime      DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`binding_id`),
  KEY `idx_target` (`target_type`, `target_id`),
  KEY `idx_tag_id` (`tag_id`),
  UNIQUE KEY `uk_target_tag` (`target_type`, `target_id`, `tag_id`, `bind_type`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='标签绑定表';
```

### 2.4 目录标签继承机制

**绑定方式**：
- 用户/AI 对目录路径直接绑定标签 → `bind_type = '0'`
- 文件位于某目录下，自动继承该目录标签 → `bind_type = '1'`

**查询某文件的所有标签**：

```sql
-- 直接绑定标签
SELECT b.tag_id, t.tag_name, t.tag_color, b.bind_type
FROM lingdoc_tag_binding b
JOIN lingdoc_tag t ON b.tag_id = t.tag_id
WHERE b.target_type = 'F' AND b.target_id = ? AND b.bind_type = '0'

UNION

-- 继承自父目录的标签（通过路径前缀匹配）
SELECT b.tag_id, t.tag_name, t.tag_color, '1' AS bind_type
FROM lingdoc_tag_binding b
JOIN lingdoc_tag t ON b.tag_id = t.tag_id
WHERE b.target_type = 'D'
  AND ? LIKE CONCAT(b.target_id, '%')
  AND b.bind_type = '0';
```

**预计算优化（可选）**：
> 若目录层级深、文件量大，可将继承标签物理写入 `lingdoc_tag_binding`（`bind_type = '1'`），避免运行时前缀匹配。文件移动时批量更新继承标签。

## 3. AI Metadata

### 3.1 一句话概括（`summary`）

- **生成时机**：文件入 Vault 后，AI 自动提取生成。
- **长度限制**：`varchar(512)`，建议控制在 100 字以内。
- **用途**：文件列表 hover 提示、检索结果摘要、AI 上下文筛选时快速判断文件相关性。

### 3.2 关键词（`keywords`）

- **格式**：JSON 数组，如 `["奖学金","操作系统","实验报告","截止时间"]`。
- **生成时机**：与 `summary` 同时由 AI 提取。
- **用途**：
  - 辅助 FULLTEXT 检索，提升检索精度
  - AI RAG 时作为文件相关性预筛选维度
  - 用户手动补充或删除关键词

### 3.3 与标签的区别

| 维度 | 标签（`lingdoc_tag`） | 关键词（`keywords`） |
|------|---------------------|---------------------|
| **来源** | 用户手动 / AI 建议 / 继承目录 | AI 自动提取 |
| **性质** | 分类维度（如"学习资料"） | 内容特征（如"奖学金"） |
| **数量** | 少而精（通常 1~5 个） | 多而散（通常 5~20 个） |
| **可编辑** | 用户可自由增删改 | AI 生成后用户可微调 |
| **继承** | 支持目录继承 | 不继承，每文件独立 |
| **检索** | 精确筛选（IN 查询） | 辅助匹配（LIKE / JSON 检索） |

## 4. 全文内容存储策略

### 4.1 问题分析

| 维度 | 估算 | 结论 |
|------|------|------|
| 空间占用 | 1000个文件 × 100KB = **100MB**；1万个文件 × 500KB = **5GB** | 万级文件场景下数据库膨胀明显 |
| 备份成本 | 大文本混入 `.ibd` 数据文件，mysqldump 单文件巨大 | 不利于冷备份与迁移 |
| FULLTEXT 维护 | 索引构建随文本量线性增长 | 大文本量下 INSERT/UPDATE 变慢 |
| 检索响应 | 100万行 × FULLTEXT ≈ 50~200ms | 可接受，但接近性能边界 |

### 4.2 推荐方案：混合分层存储

保留 `file_content` 字段，但引入**分层存储策略**，MVP 阶段暂不启用，作为预留设计：

| 层级 | 条件 | 存储位置 | 当前策略 |
|------|------|---------|---------|
| **热数据** | 文件文本 ≤ 100KB | 数据库 `file_content` | ✅ MVP 统一存数据库 |
| **温数据** | 100KB < 文件文本 ≤ 2MB | 数据库 `file_content` | ✅ MVP 统一存数据库 |
| **冷数据** | 文件文本 > 2MB | 文件系统 `{仓库}/.lingdoc/content/{file_id}.txt` | ⏸️ 预留，数据量超限时启用 |

**预留字段**：
- `content_path`：当内容存文件系统时的物理路径
- `content_size`：文本内容的字节数（用于判断存储层级）

> **MVP 阶段决策**：所有文本统一存数据库 `file_content`。当总文本量超过 **2GB** 或单文件超过 **2MB** 时，启用分层存储，将大文件内容迁移到文件系统，数据库仅保留前 5000 字符摘要。

## 5. 脱敏数据统一表

### 5.1 表结构

```sql
DROP TABLE IF EXISTS `lingdoc_desensitized_file`;
CREATE TABLE `lingdoc_desensitized_file` (
  `des_id`        varchar(64)   NOT NULL COMMENT '脱敏记录ID',
  `file_id`       varchar(64)   NOT NULL COMMENT '原文件ID（lingdoc_file_index.file_id）',
  `des_path`      varchar(512)  NOT NULL COMMENT '脱敏文件物理路径',
  `des_content`   longtext      NOT NULL COMMENT '脱敏后的纯文本内容（供AI直接读取上传）',
  `des_checksum`  varchar(64)   NOT NULL COMMENT '脱敏内容校验（SHA-256）',
  `rule_id`       varchar(64)   DEFAULT NULL COMMENT '使用的脱敏规则ID（预留）',
  `create_time`   datetime      DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`des_id`),
  UNIQUE KEY `uk_file_id` (`file_id`),
  KEY `idx_checksum` (`des_checksum`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='脱敏文件表';
```

### 5.2 业务逻辑

- **生成时机**：文件脱敏流程完成后，同步写入本表。
- **AI 上传**：优先读取本表 `des_content`，若存在则直接上传数据库中的脱敏文本，无需文件 IO。
- **数据一致性**：`lingdoc_file_index.is_desensitized = '1'` 时，本表必须存在对应记录。
- **清理策略**：删除原文件时，级联删除本表记录并清理物理脱敏文件。

## 6. 删除 `kb_document` 后的新模型

### 6.1 替代关系

`kb_document` 概念被 **两张表** 替代：
- `lingdoc_file_index`：纯文件索引（物理元数据）
- `lingdoc_file_ai_meta`：AI 处理元数据（通过 `file_id` 1:1 关联）

| 原 `kb_document` 字段 | 迁移后位置 | 说明 |
|---------------------|-----------|------|
| `doc_id` | `lingdoc_file_index.file_id` / `lingdoc_file_ai_meta.file_id` | 同一ID |
| `kb_id` | `lingdoc_file_ai_meta.kb_id` | AI 元数据表 |
| `content_text` | `lingdoc_file_index.file_content` | 文件索引表 |
| `chunk_count` | `lingdoc_file_ai_meta.chunk_count` | AI 元数据表 |
| `status` | `lingdoc_file_ai_meta.parse_status` | AI 元数据表 |
| `error_msg` | `lingdoc_file_ai_meta.parse_error` | AI 元数据表 |
| `embedding_status` | `lingdoc_file_ai_meta.embedding_status` | AI 元数据表 |

### 6.2 知识库文档判定

- `lingdoc_file_ai_meta.kb_id IS NOT NULL` → 该文件已加入知识库
- `lingdoc_file_ai_meta.parse_status = '2'` → 文本提取已完成
- `lingdoc_file_ai_meta.embedding_status = '2'` → 向量化已完成

### 6.3 关联表调整

- `lingdoc_form_reference.doc_id` → 注释改为"文件ID（`lingdoc_file_index.file_id`）"
- `lingdoc_form_field.source_doc_id` → 注释改为"来源文件ID（`lingdoc_file_index.file_id`）"
- `kb_chunk.doc_id` → 外键改为 `file_id`，指向 `lingdoc_file_index`
- `kb_embedding.chunk_id` 不变，通过 `kb_chunk.file_id` 间接关联
- `kb_knowledge_base` 表**保留**，作为知识库配置容器

## 7. 跨平台双轨配置体系

### 7.1 核心原则

1. **配置与数据库解耦**：应用级引导配置保存在本地文件系统，不依赖数据库本身。
2. **不引入 SQLite**：最终版保持 MySQL 8.0+。
3. **跨平台路径适配**：应用配置目录根据运行平台自动映射到标准用户数据目录。

### 7.2 两处 `.lingdoc/` 职责分离

| 目录类型 | 位置 | 职责 | 是否跟随仓库迁移 |
|---------|------|------|-----------------|
| **应用配置目录** | `{platform_user_data}/LingDoc/` | 应用级配置、密钥、日志、临时文件 | ❌ 不跟随 |
| **仓库元数据目录** | `{用户指定仓库根目录}/.lingdoc/` | 索引、缩略图、审计日志、版本快照、仓库策略 | ✅ 跟随仓库 |

### 7.3 跨平台应用配置目录

| 平台 | 标准用户数据目录 | 实际路径示例 |
|------|----------------|-------------|
| Windows | `%LOCALAPPDATA%` | `C:\Users\{user}\AppData\Local\LingDoc\` |
| macOS | `~/Library/Application Support/` | `~/Library/Application Support/LingDoc/` |
| Linux | `~/.config/` (XDG) | `~/.config/LingDoc/` |
| Android | `Context.getFilesDir()` | `/data/data/{package}/files/LingDoc/` |
| HarmonyOS | `Context.getFilesDir()` | `/data/app/el2/100/base/{bundle}/files/LingDoc/` |

### 7.4 配置分级存储

| 配置项 | 示例 | 存储位置 | 理由 |
|--------|------|---------|------|
| **仓库根路径** | `D:\LingDocVault` | `{platform_user_data}/LingDoc/config.json` | 启动引导配置，与数据库解耦 |
| **数据库连接** | MySQL host、port | `{platform_user_data}/LingDoc/config.json` | 启动引导配置 |
| **界面偏好** | 默认视图、主题 | `{platform_user_data}/LingDoc/config.json` | 纯客户端状态 |
| **用户运行时偏好** | 默认标签、快捷操作 | 复用 `sys_config` 表 | 与 RuoYi 兼容 |
| **仓库级策略** | 索引策略、版本保留天数 | `{仓库}/.lingdoc/config.json` | 跟随仓库走 |

### 7.5 配置读取优先级

`{仓库}/.lingdoc/config.json` > `{platform_user_data}/LingDoc/config.json` > `application.yml` 默认值
