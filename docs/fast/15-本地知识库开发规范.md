# 本地知识库开发规范

**文档编号**: LingDoc-FAST-015  
**版本**: v1.0  
**编制日期**: 2026-04-24  
**状态**: 定稿  
**配套文档**: FAST-011《Spring AI 接入评估与过渡方案》、FAST-012《Dify 集成开发规范》、SPEC-09《AI 模块架构设计》、SPEC-10《AI 模块开发规范》

> **本文档面向对象**：负责实现 LingDoc 本地知识库（RAG）功能的后端开发同学。  
> **核心目标**：为每个 Vault（文件仓库）部署一个独立的本地知识库，支持文档解析、Embedding、向量检索和 RAG 对话。

## 1. 架构概述

### 1.1 设计原则

| 原则 | 说明 |
|------|------|
| **Vault 级隔离** | 每个 Vault 拥有独立的知识库数据，跟随 Vault 迁移 |
| **配置中心化** | 知识库配置（模型、分块参数）存储在 MySQL，便于用户级统一管理 |
| **数据本地化** | 文本分块、向量、对话记录存储在 Vault 的 SQLite 中 |
| **不引入 Spring AI** | 遵循 FAST-011 结论，Spring AI 2.0 GA 前使用原生 HTTP 调用 LLM |
| **零外部依赖** | 向量存储使用 SQLite BLOB，不依赖 Milvus / Chroma / PGVector 等外部服务 |

### 1.2 数据存储架构（方案A：配置中心化 + 数据本地化）

```
┌─────────────────────────────────────────────────────────────┐
│                      MySQL 中心库                             │
│  ┌─────────────────┐  ┌─────────────────┐                   │
│  │ lingdoc_user_repo│  │ kb_knowledge_base│  知识库配置      │
│  │  (用户仓库配置)   │  │ (模型/分块参数)  │                   │
│  └─────────────────┘  └─────────────────┘                   │
└─────────────────────────────────────────────────────────────┘
                              │
                              │ 1:N（一个用户可配置多个知识库）
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    Vault 本地 SQLite                          │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐            │
│  │ lingdoc_file│ │   kb_chunk  │ │ kb_embedding│            │
│  │   _index    │ │ (文本分块)   │ │ (向量数据)   │            │
│  └─────────────┘ └─────────────┘ └─────────────┘            │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐            │
│  │lingdoc_file │ │ai_chat_     │ │ai_chat_     │            │
│  │  _ai_meta   │ │  session    │ │  message    │            │
│  └─────────────┘ └─────────────┘ └─────────────┘            │
```

### 1.3 核心数据流

```
用户上传文档到知识库
        ↓
┌─────────────────┐
│ 1. 文件保存      │  → 存入 Vault documents/ 目录
│ 2. 文本提取      │  → Apache Tika 提取纯文本
│ 3. 智能分块      │  → 按段落+固定长度+重叠分块
│ 4. Embedding     │  → 调用 Embedding API 生成向量
│ 5. 存储向量      │  → kb_chunk + kb_embedding 写入 SQLite
│ 6. 更新元数据    │  → lingdoc_file_ai_meta 更新状态
└─────────────────┘

用户发起 RAG 对话
        ↓
┌─────────────────┐
│ 1. 向量化查询    │  → 将用户问题生成 Embedding
│ 2. 向量检索      │  → SQLite 中计算余弦相似度，取 Top-K
│ 3. 上下文组装    │  → System Prompt + 检索结果 + 历史对话
│ 4. 调用 LLM      │  → 原生 HTTP POST 到 LLM API
│ 5. 流式响应      │  → SSE 返回前端
│ 6. 保存记录      │  → ai_chat_session + ai_chat_message
└─────────────────┘
```

### 1.4 与现有系统的集成点

| 集成点 | 位置 | 说明 |
|--------|------|------|
| Vault 数据源切换 | `VaultDataSourceAspect` | 拦截 `ILingdoc*Service` 方法，自动切换到当前 Vault 的 SQLite |
| Vault 上下文 | `VaultContextInterceptor` | 从 `X-Vault-Path` 请求头获取 Vault 路径 |
| 数据库初始化 | `VaultDbInitializer` | 创建/补充 Vault 的 `vault.db` 表结构 |
| 文件索引 | `lingdoc_file_index` | 知识库文档同时是 Vault 文件，共用索引 |
| AI 元数据 | `lingdoc_file_ai_meta` | 已存在，扩展 `kb_id` 关联知识库 |

## 3. 存储方案

### 3.1 向量存储：SQLite BLOB

**方案选择理由**：
- 零外部依赖，与 Vault SQLite 天然集成
- 数据跟随 Vault 迁移，无需额外导出导入
- 个人版场景下文档量可控（单 Vault 通常 < 10,000 条分块），SQLite 性能足够

**向量序列化格式**：

```java
// 存储：float[] → byte[]（IEEE 754 单精度浮点，4字节/维度）
public byte[] floatArrayToBytes(float[] vector) {
    ByteBuffer buffer = ByteBuffer.allocate(vector.length * 4);
    buffer.order(ByteOrder.LITTLE_ENDIAN);
    for (float v : vector) {
        buffer.putFloat(v);
    }
    return buffer.array();
}

// 读取：byte[] → float[]
public float[] bytesToFloatArray(byte[] bytes, int dimension) {
    ByteBuffer buffer = ByteBuffer.wrap(bytes);
    buffer.order(ByteOrder.LITTLE_ENDIAN);
    float[] vector = new float[dimension];
    for (int i = 0; i < dimension; i++) {
        vector[i] = buffer.getFloat();
    }
    return vector;
}
```

### 3.2 相似度计算：余弦相似度

在 Java 内存中计算，不依赖数据库的向量扩展：

```java
public float cosineSimilarity(float[] a, float[] b) {
    float dot = 0, normA = 0, normB = 0;
    for (int i = 0; i < a.length; i++) {
        dot += a[i] * b[i];
        normA += a[i] * a[i];
        normB += b[i] * b[i];
    }
    return dot / (float)(Math.sqrt(normA) * Math.sqrt(normB));
}
```

**检索流程**：
1. 将查询文本通过 Embedding API 生成向量
2. 从 SQLite 加载目标知识库的所有 `kb_embedding` 记录
3. 在内存中逐条计算余弦相似度
4. 按相似度降序排序，取 Top-K（默认 K=5）
5. 通过 `chunk_id` 联查 `kb_chunk` 获取原始文本

### 3.3 性能优化策略

| 策略 | 说明 |
|------|------|
| **按需加载** | 只加载当前知识库（`kb_id` 过滤）的向量，不加载全库 |
| **缓存** | 热点知识库的向量可缓存在内存（Guava Cache / Caffeine），设置 TTL |
| **分页检索** | 向量数量大时，分批次加载计算 |
| **维度裁剪** | 若使用高维模型（如 3072 维），可评估是否降维到 1536 |

### 4.2 关键接口详细规范

#### 4.2.1 上传文档到知识库

```
POST /lingdoc/ai/knowledge/doc/upload
Content-Type: multipart/form-data
X-Vault-Path: D:/LingDoc/Vault
```

**请求参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `kbId` | String | 是 | 目标知识库ID |
| `file` | File | 是 | 上传的文件（PDF/DOCX/TXT/HTML） |

**处理流程**：

```
1. 校验文件类型（白名单：pdf, docx, doc, txt, html, md）
2. 保存文件到 Vault documents/ 目录
3. 插入 lingdoc_file_index 记录
4. 更新 lingdoc_file_ai_meta：parse_status = 0（待处理）
5. 异步触发文档解析任务（线程池 / Spring @Async）
6. 返回 docId + parse_status
```

**异步解析任务**：

```
① 文本提取（Apache Tika）
   成功 → parse_status = 1（处理中）
   失败 → parse_status = 3，记录 parse_error

② 智能分块（按 kb_knowledge_base.chunk_size / chunk_overlap）
   - 优先按段落分割
   - 段落过长时按固定长度切分，保留重叠
   - 写入 kb_chunk

③ Embedding 生成（调用 Embedding API）
   成功 → 向量写入 kb_embedding，embedding_status = 2
   失败 → embedding_status = 3

④ 更新统计
   - lingdoc_file_ai_meta: chunk_count, summary, keywords
   - kb_knowledge_base: total_files, total_chunks
```

#### 4.2.2 流式对话（SSE）

```
POST /lingdoc/ai/chat/stream
Content-Type: application/json
X-Vault-Path: D:/LingDoc/Vault
```

**请求体**：

```json
{
  "sessionId": "sess_xxx",
  "kbId": "kb_xxx",
  "message": "请总结一下我的发票文件",
  "historyLimit": 10
}
```

**处理流程**：

```
1. 校验 session 存在且属于当前用户
2. 若 kbId 不为空：
   a. 将 message 生成 Embedding
   b. 向量检索 kb_embedding，取 Top-5 chunk
   c. 组装 RAG Prompt：System + 检索结果 + 历史对话 + 用户问题
3. 若 kbId 为空：
   a. 普通对话模式，不检索知识库
4. 调用 LLM API（SSE 模式）
5. 流式返回前端（逐字输出）
6. 全部返回后，保存用户消息 + AI 回复到 ai_chat_message
7. 更新 ai_chat_session 的 token 统计和最后消息时间
```

**SSE 响应格式**：

```
data: {"type":"chunk","content":"这是"}

data: {"type":"chunk","content":"回答"}

data: {"type":"done","tokenCount":153}
```

## 6. 开发约束

### 6.1 技术约束

| 约束 | 说明 |
|------|------|
| **不引入 Spring AI** | FAST-011 结论，使用原生 `RestClient` / `HttpClient` 调用 LLM API |
| **Java 17 兼容** | 不使用 Java 21 特性（Record、Pattern Matching、Virtual Threads） |
| **Jackson 3.x** | 与 Spring Boot 4 一致，包名 `tools.jackson.*` |
| **零外部向量库** | 向量存储在 SQLite BLOB，检索在 Java 内存中计算 |

### 6.2 代码规范

**Controller 命名**：

```java

@RestController
@RequestMapping("/lingdoc/ai/model")

```

**Service 命名**：

```java
public interface IAiKnowledgeService { }
public interface IAiChatService { }
public interface IAiSearchService { }
public interface IEmbeddingService { }
public interface IVectorStoreService { }
```

**数据源切换注意**：

- 知识库配置（`kb_knowledge_base`）查 MySQL：需显式切换数据源或避开 `VaultDataSourceAspect` pointcut
- 知识库数据（`kb_chunk`、`kb_embedding`）查 SQLite：Service 接口需加入 `VaultDataSourceAspect` 的 pointcut

```java
// VaultDataSourceAspect 需扩展的 pointcut（追加以下 Service 接口）
"execution(* com.ruoyi.system.service.lingdoc.IAiKnowledgeDataService.*(..)) || " +
"execution(* com.ruoyi.system.service.lingdoc.IAiChatDataService.*(..)) || " +
"execution(* com.ruoyi.system.service.lingdoc.IAiSearchDataService.*(..)) || " +
// 对应实现类也需加入
```

### 6.3 安全配置

| 项 | 要求 |
|----|------|
| 权限控制 | 所有接口加 `@PreAuthorize("@ss.hasPermi('lingdoc:ai:xxx')")` |
| API 密钥 | 存储在 `application.yml`，通过 `${}` 环境变量注入，禁止硬编码 |
| 数据隔离 | 用户只能访问自己的知识库和对话记录（SQL 中强制加 `user_id` 过滤） |
| 文件上传 | 限制文件类型和大小（单文件 ≤ 50MB） |
