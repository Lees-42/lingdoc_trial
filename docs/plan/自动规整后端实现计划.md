## 一、架构概述

采用 **Vault Inbox 模式**：文件上传后直接放入 Vault 的 `inbox/` 目录（与 `documents/` 并列），元数据存入该 Vault 对应 SQLite 的 `lingdoc_inbox` 表。AI 规整只更新建议字段，不移动文件。用户确认后，物理 move 文件到 `documents/` 下的目标目录，元数据从 `lingdoc_inbox` 移入 `lingdoc_file_index`（正式索引表）。

```
{vaultRoot}/
├── documents/        ← 正式 Vault 文件目录（syncVault 扫描此目录）
├── inbox/            ← 收件箱/临时文件目录（与 documents 并列）
└── .lingdoc/
    └── vault.db      ← SQLite 数据库（lingdoc_inbox + lingdoc_file_index）

前端上传 ──→ inbox/ 目录 + SQLite lingdoc_inbox
                ↓ organize（AI 分析，更新建议字段）
                ↓ confirm（物理 move 到 documents/ + 移入 lingdoc_file_index + 删 inbox 记录）
```

**核心优势**：文件始终只有一份；不占用双倍磁盘；未确认文件不污染正式索引表。

## 二、SQLite 表结构扩展

### 2.1 修改 `13-vault-sqlite.sql`

**物理路径**：`ruoyi-server/sql/13-vault-sqlite.sql`（需同步到 `ruoyi-framework/src/main/resources/sql/` 和 `ruoyi-system/src/main/resources/sql/`）

在文件末尾追加：

```sql
-- ----------------------------
-- 10、收件箱（Inbox）文件表
-- 存储已上传但未确认归档的文件，确认后移入 lingdoc_file_index
-- ----------------------------
CREATE TABLE IF NOT EXISTS `lingdoc_inbox` (
  `inbox_id`        TEXT    NOT NULL PRIMARY KEY,   -- 收件箱记录ID（即前端 fileId）
  `user_id`         INTEGER NOT NULL,
  `original_name`   TEXT    NOT NULL,                -- 原始文件名
  `file_type`       TEXT    NOT NULL,                -- 文件类型
  `file_size`       INTEGER NOT NULL,                -- 文件大小（字节）
  `abs_path`        TEXT    NOT NULL,                -- 文件绝对路径（ inbox/ 下）
  `status`          TEXT    DEFAULT 'uploaded',      -- uploaded/organizing/pending/confirmed/failed
  `suggested_name`  TEXT    DEFAULT NULL,            -- AI 建议文件名
  `suggested_path`  TEXT    DEFAULT NULL,            -- AI 建议保存路径（如 '工作/合同'）
  `tag_ids`         TEXT    DEFAULT NULL,            -- 建议标签ID（逗号分隔）
  `ai_summary`      TEXT    DEFAULT NULL,            -- AI 摘要
  `ai_keywords`     TEXT    DEFAULT NULL,            -- AI 关键词（逗号分隔）
  `confidence`      REAL    DEFAULT 0.00,            -- AI 置信度
  `token_cost`      INTEGER DEFAULT 0,               -- Token 消耗
  `error_msg`       TEXT    DEFAULT NULL,            -- 错误信息
  `remark`          TEXT    DEFAULT NULL,            -- 备注
  `create_time`     TEXT    DEFAULT (datetime('now','localtime')),
  `update_time`     TEXT    DEFAULT (datetime('now','localtime'))
);

CREATE INDEX IF NOT EXISTS `idx_inbox_user_id` ON `lingdoc_inbox` (`user_id`);
CREATE INDEX IF NOT EXISTS `idx_inbox_status` ON `lingdoc_inbox` (`status`);
CREATE INDEX IF NOT EXISTS `idx_inbox_user_status` ON `lingdoc_inbox` (`user_id`, `status`);

-- 触发器：自动更新 update_time
CREATE TRIGGER IF NOT EXISTS `trg_inbox_update_time`
AFTER UPDATE ON `lingdoc_inbox`
FOR EACH ROW
BEGIN
  UPDATE `lingdoc_inbox` SET `update_time` = datetime('now','localtime') WHERE `inbox_id` = NEW.inbox_id;
END;
```

> **注意**：已有 Vault 的 SQLite 数据库不会自动重建，需通过版本迁移机制或手动执行 `ALTER TABLE` / `CREATE TABLE`。首次部署建议引导用户重新初始化 Vault。

```java
package com.ruoyi.system.service.lingdoc.ai;

import com.ruoyi.system.service.lingdoc.ai.result.AiOrganizeResult;

/**
 * 自动规整 AI 服务接口
 * <p>
 * 入参：文件路径 + 文件名 + 内容（文本）+ 用户上下文（已有目录结构、标签体系）。
 * 出参：分类建议 + 标签建议 + 重命名建议 + 摘要 + 关键词。
 * <p>
 * Dify 实现：DifyAiOrganizeServiceImpl（@Primary）
 * Fallback：MockAiOrganizeServiceImpl
 */
public interface IAiOrganizeService {
    /**
     * 对单个文件进行 AI 自动规整分析
     *
     * @param fileId      文件ID（inbox_id）
     * @param filePath    文件绝对路径
     * @param fileName    原始文件名
     * @param fileContent 文本内容（非文本类型为 null）
     * @param userId      当前用户ID
     * @return 规整建议结果
     */
    AiOrganizeResult organize(String fileId, String filePath, String fileName,
                              String fileContent, Long userId);
}
```
