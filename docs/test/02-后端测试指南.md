# 后端测试指南

> 本文档面向 LingDoc 后端开发者，说明如何在本项目技术栈下编写和运行单元测试与集成测试。

---

## 1. 概述

本项目后端采用 **JUnit 5**（`junit-jupiter`）+ **Mockito** + **Spring Boot Test** 作为测试基础框架。

| 测试类型 | 目标层级 | 典型注解 | 示例类名后缀 |
|---------|---------|---------|------------|
| 单元测试 | Service / Mapper / 工具类 | `@Mock`、`@InjectMocks` | `XxxServiceTest` |
| 集成测试 | Controller（HTTP 接口） | `@SpringBootTest`、`@AutoConfigureMockMvc` | `XxxControllerIT` |

**测试目录约定**：`src/test/java` 下与被测代码保持平行的包结构。例如：

```
被测代码：ruoyi-system/src/main/java/com/ruoyi/system/service/lingdoc/LingdocVaultServiceImpl.java
测试代码：ruoyi-system/src/test/java/com/ruoyi/system/service/lingdoc/LingdocVaultServiceTest.java
```

## 3. Service 层单元测试

### 3.1 核心模式

Service 单元测试的核心思路是：**Mock 所有外部依赖（Mapper、其他 Service），只测当前 Service 的业务逻辑**。

必备注解与工具：

| 元素 | 作用 |
|-----|------|
| `@Mock` | 标记需要 Mockito 模拟的对象（如 Mapper） |
| `@InjectMocks` | 标记被测对象，Mockito 会自动将 `@Mock` 注入其中 |
| `MockitoAnnotations.openMocks(this)` | 在 `@BeforeEach` 中初始化 Mockito |
| `when(...).thenReturn(...)` | 配置 Mock 对象的返回值 |
| `verify(...)` | 验证某个 Mock 方法是否被调用 |
| `@TempDir` | JUnit 5 提供的临时目录，用于文件系统相关测试 |

### 3.2 标准模板

以 `LingdocVaultServiceTest` 为参考，新建 Service 测试时请复制以下模板：

```java
package com.ruoyi.system.service.lingdoc;

import static org.junit.jupiter.api.Assertions.*;
import static org.mockito.ArgumentMatchers.*;
import static org.mockito.Mockito.*;

import java.util.Arrays;
import java.util.List;

import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.Test;
import org.mockito.InjectMocks;
import org.mockito.Mock;
import org.mockito.MockitoAnnotations;

import com.ruoyi.system.mapper.lingdoc.XxxMapper;

/**
 * XxxService 单元测试
 */
class XxxServiceTest
{
    @Mock
    private XxxMapper xxxMapper;

    @InjectMocks
    private XxxServiceImpl xxxService;

    @BeforeEach
    void setUp()
    {
        MockitoAnnotations.openMocks(this);
    }

    @Test
    void testSelectById()
    {
        // 1. 准备数据
        Xxx expected = new Xxx();
        expected.setId("x_001");
        expected.setName("test");

        // 2. 配置 Mock
        when(xxxMapper.selectById("x_001")).thenReturn(expected);

        // 3. 执行被测方法
        Xxx result = xxxService.selectById("x_001");

        // 4. 断言结果
        assertNotNull(result);
        assertEquals("x_001", result.getId());

        // 5. 验证调用
        verify(xxxMapper).selectById("x_001");
    }

    @Test
    void testSelectById_NotFound()
    {
        when(xxxMapper.selectById("not_exist")).thenReturn(null);

        Xxx result = xxxService.selectById("not_exist");

        assertNull(result);
    }
}
```

### 3.3 验证调用链

当 Service 方法涉及多个 Mapper 的级联操作时，使用 `verify()` 确保所有预期的调用都发生了：

```java
@Test
void testDeleteWithCascade()
{
    when(xxxMapper.selectById("x_001")).thenReturn(entity);
    when(xxxMapper.deleteById("x_001")).thenReturn(1);
    when(relatedMapper.deleteByXxxId("x_001")).thenReturn(2);

    int result = xxxService.deleteById("x_001");

    assertEquals(1, result);
    verify(relatedMapper).deleteByXxxId("x_001"); // 确认级联删除被调用
}
```

### 3.4 文件系统测试（`@TempDir`）

如果 Service 涉及文件读写（如 Vault 的 `syncVault`、`createFolder`），使用 `@TempDir` 避免污染真实文件系统：

```java
@Test
void testCreateFolder(@TempDir Path tempDir)
{
    String basePath = tempDir.toString();
    int result = xxxService.createFolder("新文件夹", 1L, basePath);

    assertEquals(1, result);
    assertTrue(Files.exists(tempDir.resolve("documents/新文件夹")));
}
```

## 4. Controller 层集成测试

### 4.1 核心模式

Controller 集成测试的目标是：**启动一个精简的 Spring 应用上下文，通过 MockMvc 模拟 HTTP 请求，验证接口的输入输出、状态码和 JSON 结构**。

必备注解：

| 注解 | 作用 |
|-----|------|
| `@SpringBootTest(classes = RuoYiApplication.class)` | 加载 Spring Boot 应用上下文 |
| `@AutoConfigureMockMvc` | 自动配置 MockMvc（用于发送模拟 HTTP 请求） |
| `@MockBean` | 将指定 Service 替换为 Mock 实现，注入到 Controller 中 |

### 4.2 标准模板

以 `LingdocVaultControllerIT` 为参考，新建 Controller 集成测试时请复制以下模板：

```java
package com.ruoyi.web.controller.lingdoc;

import static org.mockito.ArgumentMatchers.*;
import static org.mockito.Mockito.*;
import static org.springframework.security.test.web.servlet.request.SecurityMockMvcRequestPostProcessors.authentication;
import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.*;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.*;

import java.util.Set;

import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.autoconfigure.web.servlet.AutoConfigureMockMvc;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.boot.test.mock.mockito.MockBean;
import org.springframework.http.MediaType;
import org.springframework.security.authentication.UsernamePasswordAuthenticationToken;
import org.springframework.test.web.servlet.MockMvc;

import com.fasterxml.jackson.databind.ObjectMapper;
import com.ruoyi.common.core.domain.entity.SysUser;
import com.ruoyi.common.core.domain.model.LoginUser;
import com.ruoyi.RuoYiApplication;
import com.ruoyi.system.service.lingdoc.IXxxService;

/**
 * XxxController 集成测试
 */
@SpringBootTest(classes = RuoYiApplication.class)
@AutoConfigureMockMvc
class XxxControllerIT
{
    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private IXxxService xxxService;

    private ObjectMapper objectMapper = new ObjectMapper();
    private UsernamePasswordAuthenticationToken auth;
    private static final Long TEST_USER_ID = 1L;

    @BeforeEach
    void setUp()
    {
        // 构造模拟登录用户（LoginUser 是 RuoYi 安全体系的核心对象）
        SysUser sysUser = new SysUser();
        sysUser.setUserId(TEST_USER_ID);
        sysUser.setDeptId(100L);
        sysUser.setUserName("admin");
        sysUser.setNickName("管理员");
        sysUser.setPassword("N/A");

        Set<String> permissions = Set.of("lingdoc:xxx:list", "lingdoc:xxx:query");
        LoginUser loginUser = new LoginUser(TEST_USER_ID, 100L, sysUser, permissions);
        auth = new UsernamePasswordAuthenticationToken(loginUser, null, loginUser.getAuthorities());
    }

    @Test
    void testGetList() throws Exception
    {
        when(xxxService.selectList(any())).thenReturn(Arrays.asList(entity));

        mockMvc.perform(get("/lingdoc/xxx/list")
                .with(authentication(auth))) // ← 必须注入认证信息
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.code").value(200))
            .andExpect(jsonPath("$.rows[0].name").value("test"));
    }

    @Test
    void testCreate() throws Exception
    {
        when(xxxService.insert(any())).thenReturn(1);

        Map<String, String> body = new HashMap<>();
        body.put("name", "test");

        mockMvc.perform(post("/lingdoc/xxx")
                .contentType(MediaType.APPLICATION_JSON)
                .content(objectMapper.writeValueAsString(body))
                .with(authentication(auth)))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.code").value(200));
    }
}
```

## 5. 安全上下文模拟详解

### 5.1 为什么必须手动注入认证信息？

本项目 Controller 大多继承自 `BaseController`，其内部通过 `SecurityUtils.getLoginUser()` 获取当前登录用户：

```java
// SecurityUtils.java（简化）
public static LoginUser getLoginUser() {
    return (LoginUser) getAuthentication().getPrincipal();
}

public static Authentication getAuthentication() {
    return SecurityContextHolder.getContext().getAuthentication();
}
```

`SecurityContextHolder` 默认从 **当前线程** 的 `SecurityContext` 中读取认证信息。在 `@SpringBootTest` + `MockMvc` 的测试模式下，每个 `mockMvc.perform()` 请求会经过 Spring Security 过滤器链，过滤器链会从请求中恢复 `SecurityContext`。

**如果请求中没有携带认证信息**，`SecurityContextHolder.getContext().getAuthentication()` 返回 `null`，`getLoginUser()` 抛出 `ServiceException("获取用户信息异常")`，测试失败。

### 5.2 正确的注入方式：`authentication()` Post-Processor

Spring Security Test 提供了 `SecurityMockMvcRequestPostProcessors.authentication()`，它会在请求进入过滤器链之前将 `Authentication` 对象写入 `SecurityContext`：

```java
mockMvc.perform(get("/lingdoc/vault/tree")
        .with(authentication(auth)))  // ← 关键
```

其中 `auth` 的构造方式：

```java
SysUser sysUser = new SysUser();
sysUser.setUserId(1L);
sysUser.setUserName("admin");

Set<String> permissions = Set.of("lingdoc:vault:list", "lingdoc:vault:query");
LoginUser loginUser = new LoginUser(1L, 100L, sysUser, permissions);

UsernamePasswordAuthenticationToken auth =
    new UsernamePasswordAuthenticationToken(loginUser, null, loginUser.getAuthorities());
```

> **注意**：`LoginUser` 的构造函数参数为 `(Long userId, Long deptId, SysUser user, Set<String> permissions)`。权限集合 `permissions` 中的字符串必须包含被测接口所需的权限码，否则接口可能因权限不足而被拦截（返回 403）。

### 5.3 常见错误：使用 `user()` 但不生效

`SecurityMockMvcRequestPostProcessors.user()` 构造的是一个简单的 `UserDetails`，而本项目要求 `Authentication.getPrincipal()` 必须是 **`LoginUser`** 类型（因为 `SecurityUtils.getLoginUser()` 会强制类型转换）。

因此，**不要**使用：

```java
// ❌ 错误：principal 不是 LoginUser，会抛 ClassCastException
mockMvc.perform(get("/xxx").with(user("admin").roles("admin")));
```

**必须**使用：

```java
// ✅ 正确：principal 是 LoginUser
mockMvc.perform(get("/xxx").with(authentication(auth)));
```

## 6. 从零开始编写新测试

### 6.1 判断测试类型

| 场景 | 推荐测试类型 | 原因 |
|-----|------------|------|
| 新增 Service 方法（业务逻辑、算法、文件操作） | **单元测试** | 快、不依赖 Spring 上下文、Mock 外部依赖即可 |
| 新增 Controller 接口（HTTP 入口、参数绑定、返回值格式） | **集成测试** | 需要验证 Spring MVC 的请求映射、JSON 序列化、权限拦截 |
| 新增工具类方法（纯函数、日期处理、字符串处理） | **单元测试** | 无外部依赖，直接断言输入输出 |
| 修改数据库表结构或 SQL | **手动验证** + **可选单元测试** | Mapper XML 的准确性更依赖 MyBatis 运行时，建议手动联调 |

### 6.2 Service 测试新建流程

**Step 1**：创建测试文件

在 `ruoyi-system/src/test/java/.../service/模块名/` 下创建 `XxxServiceTest.java`。

**Step 2**：复制模板（见 3.2 节）

**Step 3**：为每个 public 方法编写至少 2 个测试用例

- **正常场景**：输入合法参数，验证返回结果正确
- **异常/边界场景**：输入空值、不存在的 ID、越界参数，验证行为符合预期（返回 null、抛异常、返回 0 等）

**Step 4**：运行验证

```bash
cd ruoyi-server
mvn test -pl ruoyi-system -Dtest=XxxServiceTest
```

### 6.3 Controller 测试新建流程

**Step 1**：创建测试文件

在 `ruoyi-admin/src/test/java/.../controller/模块名/` 下创建 `XxxControllerIT.java`。

**Step 2**：复制模板（见 4.2 节），注意修改：
- `@MockBean` 对应的 Service 接口
- `setUp()` 中的权限集合（与被测接口的 `@PreAuthorize` 或菜单权限配置一致）
- 每个 `mockMvc.perform()` 都必须带 `.with(authentication(auth))`

**Step 3**：为每个接口编写测试

- GET 接口：验证 `status().isOk()`、`jsonPath("$.code").value(200)`、关键字段
- POST/PUT 接口：构造 JSON body，验证创建/更新成功
- DELETE 接口：验证删除成功
- 异常场景：验证 `$.code` 为 500 或 `status().is(403)`

**Step 4**：运行验证

```bash
cd ruoyi-server
# 先安装上游依赖模块
mvn install -pl ruoyi-common,ruoyi-system -am -DskipTests
# 再运行 Controller 测试
mvn test -pl ruoyi-admin -Dtest=XxxControllerIT
```

## 8. 常见问题与排障

### 8.1 "获取用户信息异常"

**现象**：运行 Controller 集成测试时抛出 `ServiceException: 获取用户信息异常`。

**原因**：MockMvc 请求未携带认证信息，`SecurityContextHolder` 中无 `Authentication`。

**解决**：每个 `mockMvc.perform()` 都必须添加 `.with(authentication(auth))`，其中 `auth` 的 principal 必须是 `LoginUser` 类型。

### 8.2 `@AutoConfigureMockMvc` 或 `@MockBean` 找不到

**现象**：编译报错 `cannot find symbol: class AutoConfigureMockMvc` 或 `class MockBean`。

**原因**：极少数构建环境下，`spring-boot-test-autoconfigure` 或 `spring-boot-test` JAR 可能缺失部分包。

**解决**：使用 fallback 方案——`MockMvcBuilders.standaloneSetup()` + Mockito `@Mock` + `@InjectMocks`。

```java
@ExtendWith(MockitoExtension.class)
class XxxControllerFallbackIT
{
    private MockMvc mockMvc;

    @Mock
    private IXxxService xxxService;

    @InjectMocks
    private XxxController xxxController;

    @BeforeEach
    void setUp()
    {
        mockMvc = MockMvcBuilders.standaloneSetup(xxxController).build();
    }

    @Test
    void testGet() throws Exception
    {
        when(xxxService.selectById(any())).thenReturn(entity);

        // standaloneSetup 不经过 Spring Security 过滤器链
        // 需要手动将 Authentication 设置到 SecurityContextHolder
        LoginUser loginUser = new LoginUser(1L, 100L, sysUser, permissions);
        UsernamePasswordAuthenticationToken auth =
            new UsernamePasswordAuthenticationToken(loginUser, null, loginUser.getAuthorities());
        SecurityContextHolder.getContext().setAuthentication(auth);

        try {
            mockMvc.perform(get("/lingdoc/xxx/x_001"))
                .andExpect(status().isOk());
        } finally {
            SecurityContextHolder.clearContext();
        }
    }
}
```

> **注意**：使用 fallback 方案时，必须在 `@AfterEach` 或 `finally` 块中调用 `SecurityContextHolder.clearContext()`，避免污染后续测试。

### 8.3 模块依赖未找到

**现象**：`mvn test -pl ruoyi-admin` 报错 `Could not find artifact com.ruoyi:ruoyi-system:jar:3.9.2`。

**原因**：兄弟模块未安装到本地 Maven 仓库。

**解决**：先运行 `mvn install -pl ruoyi-common,ruoyi-system,ruoyi-framework -am -DskipTests`，再运行测试。

### 8.4 `target/ruoyi-admin.jar` 被 JVM 锁定

**现象**：`mvn clean` 报错无法删除 `target/ruoyi-admin.jar`。

**原因**：该 jar 包正在被一个未关闭的 Java 进程占用。

**解决**：
- 检查是否有未关闭的 `java -jar ruoyi-admin.jar` 进程，关闭后再执行 `mvn clean`
- 或使用 `mvn compile` 代替 `mvn clean compile`（跳过清理阶段）

## 9. 测试命名与目录约定

| 约定项 | 规则 | 示例 |
|-------|------|------|
| 单元测试类名 | `被测类名 + Test` | `LingdocVaultServiceTest` |
| 集成测试类名 | `被测类名 + IT` | `LingdocVaultControllerIT` |
| 测试方法名 | `test + 被测方法名 + _ + 场景描述`（可选） | `testGetVaultTree`、`testGetVaultTree_Empty` |
| 目录结构 | 与被测代码包结构平行 | `service/lingdoc/` 对应 `service/lingdoc/` |
| 临时文件 | 使用 `@TempDir`，禁止写死真实路径 | — |
| Mock 对象 | 被 Mock 的类必须是接口或可被 Mockito 代理的类 | Mapper 接口、Service 接口 |
